7. 你提到过接触ReAct，为什么在Prompt里加入ReAct相关提示词，模型推理能力会变强？

   ReAct 强制模型进入「显式推理 + 工具行动 + 反馈修正」的闭环，模拟人类解决复杂问题的认知过程，从根本上弥补了大模型原生推理的三大缺陷：易幻觉、无外部信息、无自我修正。

8. 现在大模型上下文窗口越来越大，直接把大文档塞给模型提问就行，为什么还要做知识检索？

   1. 成本与延迟：暴力投喂的代价太高 💸
   虽然模型上下文变大了，但计算资源并不是无限的。
   推理成本呈线性（甚至指数）增长： 如果你每次提问都把几万字的文档全部塞进去，Token 的消耗量是巨大的。对于高频使用的企业应用，直接将整个知识库（比如几十万字的文档）作为上下文输入，其成本比只检索相关片段的 RAG 方案高出 1-2 个数量级。
   响应速度变慢： 处理 100 万 Token 的上下文，模型的推理延迟会显著上升。用户可能无法忍受每次提问都要等待几十秒甚至更久。相比之下，RAG 只检索最相关的几百个 Token，响应速度极快。
   2. “中间迷失”：模型也会“注意力涣散” 🧠
   这是长上下文模型目前最大的软肋。
   注意力分布问题： 研究表明，当上下文过长时，大模型往往会出现“中间迷失”（Lost in the Middle）现象。模型倾向于关注开头和结尾的信息，而忽略中间的内容。
   信息淹没： 如果文档中包含大量无关的“噪声”，关键信息很容易被淹没。模型在处理超长文本时，注意力会被分散，导致回答质量下降，甚至出现幻觉。
   RAG 的优势： RAG 的作用就像是“划重点”。它先筛选出最相关的片段，去除了噪声，让模型在一个短小精悍的上下文中进行推理，准确率反而更高。
   3. 实时性与数据更新：模型不是数据库 🔄
   静态与动态： 大模型的上下文窗口是“一次性”的。如果你把文档塞进上下文，模型处理完这次对话后，这些信息就“忘”了（除非你每次都重新塞）。
   数据更新难题： 企业的知识库是实时变化的。如果采用“全文投喂”，一旦某份文档更新，你就需要重新处理整个庞大的上下文。而 RAG 系统可以独立维护向量数据库，文档更新只需更新索引，不影响整体架构。

10. 知识库平台用TRAG向量数据库+ES做多路召回，为什么要这么做？

    向量数据库：擅长“语义理解”
    优点：它能理解词语背后的含义，而不仅仅是字面匹配。例如，当用户提问“如何修改登录密码”时，即使文档中没有“修改”这个词，而是“重置密码”或“找回账户”，向量检索也能找到相关内容。
    缺点：在处理精确匹配时表现不佳。对于产品型号、合同编号、SKU等包含特定数字和字母组合的查询，向量检索容易出错或完全忽略，向量检索可能会返回一些语义上“相似”但完全错误的知识。
    Elasticsearch (ES)：擅长“关键词匹配”
    优点：它基于倒排索引，能非常精确、快速地匹配关键词。对于查询“iPhone 15 Pro Max 256G”或“合同编号A-2023-019”这类包含精确信息的请求，ES是最佳选择。
    缺点：它无法理解语义。如果用户用“高端手机”来搜索，而文档中只写了“旗舰手机”，ES就可能无法召回这条相关文档。

18. 从你的角度看，为什么要用多个Agent实现功能？

     1. 解决“Prompt 过载”与指令遗忘问题（最致命的问题）
    - 单 Agent 的困境 ：如果用一个 Agent，你必须在一个 Prompt 里塞入所有的要求：“你要扮演 Osborn，记住这段背景故事，提取长期记忆，分析玩家的情绪，推测玩家的意图，还要注意语气，并且不能违反伦理约束……”。当指令太长、太复杂时，大模型会出现严重的**“注意力失焦”（Lost in the middle）**，它可能记住了人设，却忘了分析情绪；或者分析了意图，却偏离了说话风格。
    - 多 Agent 的优势（职责单一） ：通过解耦，每个 Agent 只做一件事，Prompt 非常短且聚焦。
     2. 极大地提升了“可解释性”与“易调试性” (Observability)
    - 单 Agent 的困境 ：如果 NPC 突然说了一句莫名其妙的话，你很难知道是哪个环节出了问题（是没记住背景？是情绪算错了？还是意图理解错了？）。
    - 多 Agent 的优势（白盒化） ：在 NpcState 中，每个 Agent 都会留下自己的“思考痕迹”（如 contextual_thoughts , tom_hypothesis_thought , subconscious_response ）。当你进行 Debug 时，你可以清晰地看到整个信息流的传递过程，精准定位是哪个 Agent（哪个认知模块）出了问题并单独优化它的 Prompt。
     3. 模型路由与成本优化
    - 单 Agent 的困境 ：为了保证所有复杂任务都不出错，你可能必须全局使用最昂贵、最慢的模型（如 GPT-4o 或 Claude-3.5-Sonnet）。
    - 多 Agent 的优势 ：你可以为不同的 Agent 配置不同的模型。例如，简单的“记忆提取”或“情绪打分”可以使用便宜快速的小模型；而核心的“ToM 意图推断”和“最终话语生成”再调用逻辑能力最强的昂贵大模型（例如你打开的配置里使用的 venus-deepseek-v3.2 ）。这能在保证效果的同时大幅降低 API 成本和响应延迟。

26. 调用模型时可能出现高延迟、失败、超时，工程上怎么做兜底？

    🛡️ 熔断与降级：快速失败，保护系统
    当模型服务出现问题时，首要目标是避免请求长时间阻塞，拖垮整个系统。
    熔断器 (Circuit Breaker)
    原理：熔断器如同电路中的保险丝，有三种状态：
    闭合 (Closed)：请求正常通过。
    打开 (Open)：当失败次数达到阈值，熔断器打开，后续请求会立即失败，不再调用有问题的服务，实现“快速失败”。
    半开 (Half-Open)：经过一段等待时间后，允许少量试探性请求通过。如果成功，则熔断器闭合；如果失败，则继续保持打开状态。
    作用：防止因单个模型服务故障导致系统资源（如线程池）耗尽，引发雪崩效应。
    服务降级 (Service Degradation)
    原理：当熔断器被触发或服务调用失败时，不直接返回错误，而是执行一个预设的降级逻辑。
    降级策略：
    返回默认值：返回一个友好的提示信息，如“服务繁忙，请稍后再试”。
    返回缓存结果：返回之前缓存的、相对陈旧但可用的数据。
    调用轻量模型：切换到响应更快、成本更低的轻量级模型来处理请求。
    🔄 多模型冗余：主备切换，保障可用性
    不要将鸡蛋放在一个篮子里。通过配置多个模型服务，可以在主模型不可用时自动切换。
    设计降级链路：构建一个模型优先级队列，例如 ['gpt-4o', 'claude-3', '本地模型']。
    自动故障转移：当主模型（如 GPT-4o）调用失败或触发熔断时，系统自动尝试调用备用模型（如 Claude-3）。如果备用模型也失败，可以继续尝试下一个，直到成功或所有模型均不可用。
    智能调度：可以根据请求类型、成本和性能要求，智能匹配最优模型。例如，将复杂的代码生成任务分配给强大的模型，而简单的问答则交给成本更低的模型。

27. 你了解ReAct和MCP，从你的角度理解，为什么需要MCP？

    1. 打破信息孤岛
    AI模型本身的知识库是静态的（训练数据截止于某个时间点），且无法访问私有数据。
    问题：你想让AI帮你查询公司内部的数据库、实时股价、今天的气温或你网盘里的文件。传统模型做不到。
    MCP的解法：MCP定义了标准化的“资源”接口。开发者可以构建一个MCP服务器，向外暴露数据库、API、文件系统等资源。任何支持MCP的AI客户端（如IDE、聊天机器人）都能通过同一套协议访问这些资源，让AI模型“看到”实时或私有数据。
    2. 统一工具调用接口
    AI模型需要调用外部工具来执行操作，比如发送邮件、创建日程、修改代码、控制物联网设备。
    问题：每个工具都有自己的API（例如，Gmail的API、Slack的API、Jira的API）。每接入一个新工具，开发者都需要为每个AI模型编写特定的集成代码。这是巨大的重复劳动。
    MCP的解法：MCP定义了标准化的“工具”接口。开发者只需将一个工具封装成MCP服务器（实现标准的“列出工具”、“调用工具”方法），任何MCP客户端都能发现并使用它。“一次封装，处处使用”。
    3. 增强安全性与可控性
    让AI模型自由访问系统和数据存在巨大风险。
    问题：恶意提示词可能诱导模型删除文件、泄露隐私。用户无法精细控制AI能做什么、不能做什么。
    MCP的解法：MCP将AI模型（客户端）与操作执行端（服务器）明确分离。服务器可以独立实现细粒度的权限控制（如“只读”、“需要用户审批”）。模型只是提出工具调用请求，真正的操作由受信任的MCP服务器在受控环境中执行。这提供了一个清晰的安全边界。

29. 你怎么理解Agent？

    Agent（智能体 / 人工智能体），是指能感知环境、自主思考、主动决策、调用工具、执行行动，并持续学习优化以完成特定目标的智能系统或实体。

    | 层级 | 核心职责 | 关键组件/技术 |
    | 1. 感知层 | 输入与解析：接收用户指令、环境状态或多模态信息（图片/语音）。 | 自然语言理解、传感器数据对齐、提示词构建。 |
    | 2. 记忆层 | 存储与检索：记住用户偏好、历史对话和任务状态。 | 短期记忆（上下文窗口）、长期记忆（向量数据库/RAG）、记忆压缩。 |
    | 3. 决策/规划层 | 大脑与推理：理解目标，拆解任务，规划路径，自我反思。 | LLM 核心引擎、推理范式（CoT/ReAct）、任务规划算法。 |
    | 4. 工具/执行层 | 行动与操作：调用外部能力完成具体工作。 | API 调用、代码解释器、浏览器控制、Function Calling。 |
    | 5. 编排与控制层 | 流程管理：管理 Agent 的生命周期、多智能体协作。 | 状态机、工作流引擎（DAG）、多 Agent 协作拓扑（如主从模式）。 |
    | 6. 安全与对齐层 | 护栏与合规：确保行为可控、不越权、内容安全。 | 权限沙箱、输出过滤、人类介入点（Human-in-the-Loop）。 |

35. 创建 Graph 实例后，怎么提交到后端并调用模型返回结果？

    在 app.py 中通过 agent = OsbornGraph(...) 实例化 Graph 后，整体的调用流转过程如下：

    1. 图的编译绑定 OsbornGraph 继承自 NpcGraph 。在它初始化时，会依次实例化所有的子 Agent（如 ContextualPerceptionAgent , TomInferenceAgent 等）。最后调用 build_graph() 方法，通过 StateGraph 把上述的各个处理节点（Node）和条件边（Conditional Edges）串联起来，并调用 graph.compile() 编译成可执行的工作流实例。

    2. 接收请求与状态加载 当用户发送消息时，前端会触发 app.py:L76-L82 中的 chat_with_detail 协程，此时代码调用了图的执行入口 agent.invoke_stream(...) 。
    在该方法内部，系统会通过 load_state 方法加载历史对话记录，将用户的最新消息装配到初始化的 NpcState 中。

    3. 图流转与模型真正调用 随后，系统调用 LangGraph 原生的 self.graph.astream(state) 开始流式执行。状态数据会在图的各个节点中流转。
    在流转到具体节点（如 contextual_perception ）时：

    ```
    output = await self.
    contextual_perception_agent.process(**inputs)
    return {"contextual_thoughts": [output]}
    ```
    这里的 process 方法内部，才是 真正将组装好的 Prompt 提交给后端大模型，并发起 LLM API 请求的地方 。模型返回的 output 会被包装成字典形式返回，LangGraph 接着自动将其更新覆盖到总体的 NpcState 中，然后向下游节点继续传递。

    4. 结果的流式返回 在 app.py 中，通过 async for chunk_type, chunk_content in response: 遍历上述执行过程的产出。
    如果当前运行在 debug 模式，前端会捕获各个节点的中间思考结果（例如记忆检索的结果、约束反思的具体建议等），利用 yield 即时发送到 UI 界面展示。当图流转到最后的 final_utter 节点时，提取出最终纯净的台词并作为结果返回给玩家。

38. Redis大Key问题

    核心方案：拆分大 Key
    这是最根本的解决方法，将一个大 Key 拆分成多个小 Key。
    String 类型：将一个超大的 JSON 字符串按业务字段拆分。
    拆分前：SET user:1001:profile "{...大量数据...}"
    拆分后：SET user:1001:profile:base "..."、SET user:1001:profile:detail "..."
    Hash/List/Set 类型：对集合内的元素进行分片。例如，对一个包含 100 万元素的 List，可以按 ID 取模，拆分成 100 个包含 1 万元素的小 List，如 list:shard:0 到 list:shard:99。
    辅助方案：数据压缩
    对于无法拆分或拆分成本过高的 String 类型数据（如长文本、JSON），可以在存入 Redis 前使用 Gzip、Snappy 等算法进行压缩，读取时再解压。这能有效减少内存占用和网络传输量。

44. Harness Engineering是什么

    Harness Engineering构建了一套包裹在AI模型外的工程外壳，通常包含四个层次：
    记忆层：通过项目根目录下的CLAUDE.md、AGENTS.md等“规则文件”，持续向AI注入项目的核心信息（如技术栈、代码规范、项目目标），确保其大方向不跑偏。
    执行层：赋予AI调用外部工具的能力，如执行终端命令、读写文件、使用浏览器等，让AI能从“思考者”变为“行动者”。
    反馈层：通过让AI执行单元测试、代码检查（Lint）等自动校验，并将运行报错或测试失败信息回传，形成闭环，驱动AI进行自我修正。
    编排层：将一个大任务拆解为多个有明确标准和依赖关系的子任务，让AI像遵循项目计划一样分步执行，防止其在循环中陷入混乱或死循环。

49. 如何写好一个 Skills？有什么原则？

    要写出“好用”的 Skill，而非仅仅“能用”，你需要遵循以下核心原则：
    原子性 (Atomicity)：坚持单一职责原则。每个 Skill 应该像一块积木，只专注于一件事，并把它做好。小而美的 Skill 更易于复用和组合成复杂的工作流。
    示例优于解释 (Few-Shot Prompting)：与其用大量文字描述，不如直接给出几个清晰的输入和输出示例。榜样的力量是无穷的，AI 能通过具体例子快速理解你想要的格式、风格和行为。
    结构化指令 (Structured Instructions)：
    定角色：为 AI 设定一个明确的专家人设，例如“你现在是一位资深的代码审查员”。
    拆步骤：将任务流程分解为一步步的具体指令，引导 AI 按顺序“思考”。
    画红线：明确告知 AI“不能做什么”，设定清晰的边界，防止其产生“幻觉”或执行危险操作。
    接口化设计 (Interface Design)：像设计软件 API 一样，明确定义 Skill 的输入参数和输出格式（例如固定输出 JSON 或 Markdown）。这能让你的 Skill 被其他程序或其他 Skill 稳定地调用和集成。
    迭代优化 (Iterative Refinement)：将 Skill 视为一个持续进化的产品。在实际使用中，留心那些不尽如人意的“失败案例”（Bad Cases），并将它们转化为新的规则或反例，补充到 Skill 的定义中，让它越来越聪明、可靠。

     Skill 名称：xxx

     触发条件
    - 用户提到哪些关键词/场景时启用

     角色与目标
    - 你扮演什么角色
    - 你要达成什么具体结果

     工作流程
    1. 第一步：收集信息（需要用户提供什么）
    2. 第二步：分析方法/规则
    3. 第三步：输出格式（必须严格遵循）

     输出模板（示例）
    [给 1-2 个具体输入输出示例]

     约束与边界
    - 什么情况下不处理/拒绝
    - 不能做哪些事

     异常处理
    - 信息不足时怎么办（反问用户？给默认值？）

51. KV Cache是什么

    KV Cache（键-值缓存）是一种用于加速大型语言模型（LLM）推理过程的关键优化技术。
    简单来说，它通过“以内存换算力”的方式，将模型生成文本时已计算过的中间结果缓存起来，避免在生成下一个词时进行大量重复计算，从而显著提升推理速度。
    🤔 为什么需要KV Cache？
    要理解KV Cache，首先要了解大模型的生成方式。大模型是“自回归”的，这意味着它像一个打字员，一次只生成一个词（Token），并且生成下一个词需要参考前面所有的词。
    在没有KV Cache的情况下，模型每生成一个新词，都需要把从第一个词到当前词的所有内容重新计算一遍。这就像你每写一个新句子，都要把整篇文章从头到尾重读一遍，效率极低。随着文章越来越长，计算量会呈指数级增长，导致生成速度越来越慢。
    💡 KV Cache如何工作？
    KV Cache的解决方案非常巧妙：
    首次计算并缓存：当模型处理初始输入（如你的提问）时，它会计算每个词对应的键（Key）和值（Value）向量，并将这些结果存储在名为“KV Cache”的高速内存中。
    复用缓存：当模型要生成第一个回答的词时，它只需要计算这个新词的查询（Query）向量，然后直接去KV Cache中读取之前所有词的键（K）和值（V）向量，用于注意力机制的计算。
    更新缓存：新生成的词也会被计算成新的K和V向量，并添加到KV Cache中，供生成下一个词时使用。
    这个过程可以形象地理解为：
    无KV Cache：每写一个新字，都要把整篇文章重抄一遍。
    有KV Cache：文章抄一遍存起来，之后每写一个新字，只需参考已存好的文章，并把新字也加进去。

52. Langgraph里面的核心组件有哪些

    🧠 状态 (State)
    状态是 LangGraph 中所有节点共享的全局数据中心，可以理解为整个工作流的“统一记忆中枢”。
    作用：它负责存储任务执行过程中的所有关键信息，例如用户输入、对话历史、中间推理结果、工具调用记录、任务进度等。
    ⚙️ 节点 (Node)
    节点是 LangGraph 中的最小功能单元，是承载具体业务逻辑的“任务执行器”。
    作用：每个节点都是一个独立的 Python 函数（同步或异步），负责执行一项具体的任务。
    🔗 边 (Edge)
    边定义了节点之间的流转规则，是控制工作流执行路径的“导航器”。
    作用：它决定了在当前节点执行完毕后，下一个应该执行哪个节点。

53. A2A 协议的关键组件及其作用

    Agent Card：让代理“自己打广告”——谁我是、我在哪里（URL）、我能做什么（技能描述）、我需要什么认证（认证方式）。客户端代理看到这个就知道能不能用这个代理。
    Client/Remote Agent：角色区分很重要：客户端发起、远端执行。但在实际中一个代理可能同时扮演两种角色（即既能发起任务也能被发起者调用） 。
    Task：是协作的核心。一次任务可能只是一个简单请求，也可能是一个包含多个步骤、要等待较长时间的复杂流程。A2A 支持 “长任务” 的状态追踪。
    Message/Part：细化任务中交互的载体。因为代理可能需要交换各种模态（文本、文件、图像、输入表单等），所以 Part 则是协议定义的最小交换单元。
    Artifact：任务的结果。客户端代理收到 Task 完成信号后，从 Artifact 中取得成果。
    Discovery/Capabilities：在多代理生态中很关键。客户端需要在众多代理中挑选合适者，这就依赖于能力描述与发现机制。
    Transport/Protocol：协议说明了通信的方式和规则。使用标准通信方式可降低集成难度。
    Auth/Authorization：企业级使用中安全不能忽视，A2A 在设计中明确将其作为核心原则。

55. 知识清洗

      1. 纯文本类解析（TXT）
         文本解析（如 txt_parser.go ）的清洗主要集中在格式化和空白符处理：

    - 编码清洗 ：解析前会先检测文件编码，自动将 GB-18030 和 UTF-16LE 等非常规编码转换为标准的 UTF-8。
    - 去除回车符 ：将文本中所有的 \r 字符强制替换为空： txt_parser.go:L155
    - 去除首尾空白与空切片过滤 ：在执行两级分块（按 \n\n 分块、按标点分句）时，会对每个片段执行 strings.TrimSpace ，并且会直接丢弃清洗后为空的无效切片： txt_parser.go:L79-L81

      2. 表格类解析（Excel / CSV）
         表格数据（核心逻辑在 excel_parser.go 中复用）的清洗主要针对表头标签、换行和特殊字符：
    - 表头特殊字符清洗 ：强制去除列名中可能附带的不可见字符（如 BOM 头 \uFEFF ）： excel_parser.go:L92
    - 表头元数据标签清洗 ：在组装知识内容前，会将配置的 [索引+知识] 、 [索引] 、 [无列名] 、 [元数据] 等标志性 tag 从列名中移除，避免污染最终生成的知识文本： excel_parser.go:L202-L204
    - 单元格换行清洗 ：在非 IsUcBase 模式下，默认会把单元格内容里的换行符 \n 全部替换为空： excel_parser.go:L206

56. python编程时什么时候选择多进程，什么时候选择多线程，从IO密集和计算密集的角度考虑

    在Python编程中，选择多进程还是多线程，核心决策依据就是任务的类型：是 I/O密集型 还是 计算密集型。
    这个选择的根本原因在于 Python 的一个关键特性：全局解释器锁（GIL）。GIL 确保同一时刻只有一个线程在执行 Python 字节码，这使得 Python 的多线程无法在多核 CPU 上实现真正的并行计算。

    为什么选择多进程？
    因为多进程可以绕过 GIL 的限制。每个进程都拥有自己独立的 Python 解释器和内存空间，因此也拥有独立的 GIL。这意味着多个进程可以被操作系统分配到不同的 CPU 核心上，实现真正的并行计算，充分利用多核 CPU 的性能优势。

    为什么选择多线程？
    当线程执行 I/O 操作（如发起网络请求或读取文件）时，它会因为等待而阻塞。在这个等待期间，Python 解释器会自动释放 GIL，让其他线程有机会获取 GIL 并执行。
    这样，虽然同一时间只有一个线程在运行，但通过在 I/O 等待间隙高效地切换任务，程序的整体吞吐量和响应速度得到了极大的提升。此外，线程比进程更轻量，创建和切换的开销都小得多，非常适合处理大量并发的 I/O 任务。

57. trag查询优化

    通过对比最早的代码与现在的 HandleRecallSearch 实现，可以看出优化主要集中在 ES (Elasticsearch) 和 TRAG (向量检索) 的请求构建与并发调用模型上。
    以下是对这个优化点的深度分析：

    核心优化点解析
    - 老版本的痛点（N次请求放大） ：在老代码的 BuildSearchRequest 中，当用户传入 [ o bj ec tO bj ec t ] N 个 knowledgeBaseIds 时，系统会循环遍历这 [ o bj ec tO bj ec t ] N 个 ID，为每一个知识库分别构建独立的 [es, trag] 引擎请求对象。随后交由底层并发执行，这意味着会向 TRAG 引擎发起 [ o bj ec tO bj ec t ] N 次独立的 HTTP POST 请求 。
    - 新版本的改造（合并为1次请求） ：在新版的 BuildSearchRequest 中， 仅仅构建了 ES 引擎的请求 ；而对于 TRAG 引擎，新引入了 TragBatchSearchFromMultiKnowledgeBase 批量检索方法。该方法会把 [ o bj ec tO bj ec t ] N 个知识库对应的 CollectionCodes 提取并合并成一个数组，将它们封装进 1 个 TRAG 请求体中发送给远端模型服务。
    为什么要这样考虑？（老版导致资源占用的根本原因）
    - 重复的 Embedding（向量化）计算消耗 ：TRAG 引擎是基于向量（Dense Vector）检索的。用户输入了一句 req.Keyword （比如“如何强化装备”），TRAG 服务需要先调用大模型将这句话转换成向量。在老版代码中，发送了 [ o bj ec tO bj ec t ] N 个知识库检索请求，TRAG 服务器收到后，会对 同一句话 进行 [ o bj ec tO bj ec t ] N 次重复的向量化计算 。GPU/计算资源被这些完全冗余的 Embedding 操作极大地浪费了，这就是你提到“占用了测试环境很多TRAG计算资源”的根本原因。
    - 过多的网络 I/O 与连接数瓶颈 ：原本每个请求需要单独走 HTTP 协议。如果同时查 50 个知识库，就要瞬间建立 50 个 TCP 连接。这不仅会阻塞 Go 语言侧的 Goroutine（因为引擎内部有 semaphore := make(chan struct{}, 5) 限制了并发为 5，导致排队等待），还会打满 TRAG 服务的连接池或触发限流。

60. 怎么让大模型稳定输出json

    在大模型生产落地中，要让模型稳定输出JSON格式，核心不能只靠提示词，得用四层工程化方案层层保障，这也是解决大模型输出格式不可控的关键方法。
    首先是提示词优化，不只是提要求，还要给标准JSON示例让模型模仿，减少它的随意发挥；
    其次利用模型原生API能力，开格式强制模式或用函数调用，提前定义字段类型来约束结构；
    然后如果是本地部署模型，做底层生成约束，在词生成阶段拦截不符合JSON语法的字符，从源头规范；
    最后是兜底容错机制，拿到结果后用代码做格式校验，出错就把问题回传给模型让它修正重生成。
