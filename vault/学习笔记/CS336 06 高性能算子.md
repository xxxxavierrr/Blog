1. 为什么需要线程块
	1. 块内线程可使用共享内存
	2. 速度接近L1 cache
	3. warp：减少控制单元（能够一次调度一组线程）
2. 调度目标
	1. 所有SM执行线程块：1 wave
	2. 最后的wave会有空闲SM
	3. block 最好能整除SM（一般四倍以上）
3. 计算密集型
	1. FLOP >内存移动字节
	2. 经过优化后，瓶颈在内存上，尽量减少
4. 基准测试
	1. 热身：避免测量的是机器码加载等时间
	2. torch.cuda.synchronize()：确保CPU、GPU同步运行，相互等待，准确计算时间
	3. benchmark("sleep", lambda : time.sleep(50 / 1000))  （sleep为实际操作指代）
5. 性能分析
	1. 慢函数定位、查看CUDA底层实际执行
	2. [PyTorch Profiler — PyTorch Tutorials 2.12.0+cu130 documentation](https://docs.pytorch.org/tutorials/recipes/recipes/profiler_recipe.html)
	3. 矩阵加法：底层调用aten:add CPU调用CUDA执行 最终时间99:1
	4. 矩阵乘法：aten:matmul  aten:mm，dim=128时1:1
		1. dim不同，底层核函数、乘法原语不同
	5. Nvidia Nsight System
	6. MLP运行14s
		1. 前7.5s加载库
		2. ~9.5s构建模型
		3. layer0热身：JIT，非常慢
		4. CPU layer与GPU错位：CPU将命令排队发送给GPU（提前发送命令有上限）
		5. layer间打印loss：CPU必须等待GPU计算完，无法提前发送
			1. 导致CPU空转，CPU瓶颈
		6. CPU与GPU脱钩：能够使用高级语言的同时（CPU稍慢），充分使用GPU
6. 算子
	1. 以 GELU 为例，原生 PyTorch 拆分多步独立算子：
		1. x → matmul → add → tanh → mul → mul
		2. 每一步写回全局显存
	2. 算子融合：
		1. 中间结果存SM
		2. 仅输入输出操作全局显存
	3. CUDA
		1. 核函数：
			1. global函数传入block ID， block dim，thread ID
			2. 线程计算所在位置，超出张量长度直接 return
			3. 做实际运算
		2. CPU调用：
			1. 判断device在CUDA
			2. 判断连续存储
			3. 给输出分配地址：empty_like(x)
			4. 传num_block，block_size，num_elements，输入输出地址
		3. 完全可控内存读写
	4. Triton
		1. 核函数：
			1. 与CUDA操作单个线程不同，Triton操作整个block
			2. offsets = pid * BLOCK_SIZE + triton.arange(0, BLOCK_SIZE) 
			3. mask确保offsets<张量长度
			4. 加载内存（向量序列）->计算->写入
	5. torch.compile：自动融合算子
