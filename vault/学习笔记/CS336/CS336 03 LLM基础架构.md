3. 激活器
	1. GeLU：GELU(x)=x⋅Φ(x)
		1. 连续可导
		2. 不直接丢参数，改为缓慢趋近0
		3. ![[CS336 03 LLM基础架构-3.png|438]]
	2. GLU（门控）：
		1. max(0,xW1​)→max(0,xW1​)⊗(xV)
		2. ⊗：逐元素乘
		3. 控制分支xV，判断哪些特征保留/弱化，比传统ReLU更灵活
	3. GeGLU：GELU(xW)⊗(xV)
	4. SwiGLU：Swish(xW)⊗(xV)
		1. Swish(x) = x * Sigmoid(x)
	5. 将门控参数中W参数缩小到原来2/3，以保证整体参数量一致

7. softmax稳定性
	1. z-loss
		1. 训练后期，logit z 爆炸，softmax偏向极端
		2. 采用z-loss稳定logit z
	2. qk norm
		1. q、k越来越大，score爆炸， softmax极端
		2. 计算score之前做归一化，收敛参数大小
8. 注意力头
	1. GQA/MQA
		1. 计算密度：每次内存取数进行的计算量
		2. FLOPs：O (bnd2) 
		3. 内存访问量：O (bnd + bhn2 + d2)
			1. bnd：QKV
			2. bhn2：QK得到的score
			3. d2：x到QKV的参数
		4. KVCache:
			1. 存旧token的KV
			2. 每次只用重新生成最新token的QKV
			3. FLOPs：O (bnd2) 
			4. 内存访问量：O (bn2d + nd2) ？
			5. 计算密度比原来差
		5. MQA：Q 多头、K/V 单头、所有 Q 共用同一套 KV
		6. GQA：多 Q 多 KV（分组共享）
		7. MLA：KV表示为低纬度张量C，使用时再计算
	2. 滑动窗口注意力
		1. 采用3个RoPE+滑动窗口/1个NoPE+多头为1组
9. 算子
	1. 三要素：
		1. Grid: collection of thread blocks: numBlocks = (2, 4), blockDim = (1, 8)
		2. Thread block: collection of threads: blockIdx = (0, 1)
		3. Thread: single unit of operation: threadIdx = (0, 3)
	2. 调试：os.environ["CUDA_LAUNCH_BLOCKING"] = "1"
	3. CPU triton_gelu()  -> GPU triton_gelu_kernel()
	4. CPU:
		1. 确保张量在GPU
		2. 确保张量连续存储
		3. 结果分配内存：torch.empty_like()
		4. 确定网格数：张量元素数量/线程块大小
		5. 启动核函数
			1. 传参：num_blocks、block_size、输入输出指针、num_elements
	5. GPU:
		1. __ global __
		2. 获取当前坐标：block ID * block 维度 + thread ID
		3. 确保坐标 < 元素总数
		4. 进行计算
	6. 编译加载到py
		1. module = load_inline(cuda_sources, cpp_sources)
10. triton：
	1. 自动管理突发模式（合并调用）
	2. 共享内存管理
	3. SM内线程调度
	4. SM之间仍需手动管理
	5. 核函数：
		1. 无thread ID
		2. 块内坐标offsets = block_start + tl.arange(0, BLOCK_SIZE) 
		3. 所有线程offset一次性处理为向量
		4. mask = offsets < num_elements
		5. x = tl.load(x_ptr + offsets, mask=mask) 读取一批输入
	6. cuda写单个线程，triton写线程块（向量化）

12. 规约
	1. 计算mean/sum，block为一列
	2. 不用规约时，block size 更多是在说“计算网格怎么切 tile”；规约时，block size 常常是在说“每次攒多少元素来归并成更少的结果“
