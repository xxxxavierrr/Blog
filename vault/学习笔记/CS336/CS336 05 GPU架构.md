## 结构
1. 与CPU相比，计算单元更多，控制逻辑更少
	1. CPU：低延迟，尽快完成单个任务
	2. GPU：高吞吐量，尽快完成所有任务
2. 原子单元：流式多处理器SM
	1. 包括多个流式处理器SP，每个SP并行执行大量线程
	2. SM控制，SP接受相同命令处理不同数据
3. 内存
	1. 内存离SM越近，速度越快
		1. L1、共享内存（SM内部）
		2. L2（紧挨SM）
		3. 芯片外DRAM（显存）（全局内存）
	2. ![[CS336 05 GPU架构.png|379]]
4. 运行
	1. Blocks：线程块，调度单元固定分配到一个SM ，Block：SM=N：1
	2. Warp：线程束，最小硬件单元
	3. Thread：线程，最小软件单元

## 计算速度
1. 屋顶线模型
	1. ![[CS336 05 GPU架构-1.png|529]]
	2. 横轴：计算密度，纵轴：硬件负载
	3. 斜线：内存受限区，计算密度不足，横线：算力受限区，硬件能力不够
	4. 菱形：稠密，圆形：稀疏
2. 如何让GPU运行更快：
	1. 条件语句
		1. 单指令多模型：warp执行相同指令
		2. 条件语句可能造成线程交替运行，其他线程休眠
	2. 低精度
		1. Relu：x=max(x，0)
			1. fl32：
				1. 1 read (x), 1 write (if x < 0)，8bytes
				2. 1 comparison，1 FLOP
				3. 8bytes/FLOP
			2. fl16/bf16：
				1. 1 read (x), 1 write (if x < 0)，4bytes
				2. 1 comparison，1 FLOP
				3. 4bytes/FLOP
			3. 内存占用减少一半
	3. 融合操作
		1. ![[CS336 05 GPU架构-2.png|467]]
		2. pytorch compile自动优化简单的融合操作
	4. 重计算
		1. 牺牲计算量避免内存访问
		2. ![[CS336 05 GPU架构-3.png|367]]
		3. 重新计算前向传播来获取更少的内存操作
	5. 内存合并访问
		1. 显存访问模式：突发模式
		2. DRAM 存储单元是电容，电容电荷信号极微弱，必须靠灵敏放大器把微小电压差放大成标准高低电平，才能读出数据
		3. 读取任意数据都会激活一行，若一个warp内32线程同时访问，可一次取走32位
		4. 目的：分摊固定行激活 / 放大的延迟成本
		5. 矩阵计算中，若数据按行在内存中存储，最好按列遍历，这样每一次遍历都会同时取一整行，可以减少访问内存次数
	6. 分块
		1. 目标：从显存移到共享内存来减少访问显存次数
		2. ![[CS336 05 GPU架构-4.png]]
			1. 不使用分块时，不同线程多次从DRAM加载同一个浮点数
			2. 不使用分块时，N矩阵被迫从一列取数据，内存地址不连续
			3. Block里的32x32线程，将32x32数据分别从M、N矩阵搬入SM内共享内存，之后计算，能够避免块内重复加载、实现合并访问
		3. 分块大小影响因素：
			1. 合并内存访问
			2. 共享内存大小
			3. 均匀划分，防止出现过小的块导致SM闲置
		4. 分块和合并内存访问未对齐：
			1. 易造成多次内存访问
			2. 解决方法：padding填充矩阵，为线程大小整数倍
	7. ![[CS336 05 GPU架构-5.png|475]]
		1. K越大，FLOPS越大：分块K越大，共享内存中一次放入的数据越多，减少拉取相同块的次数，数据复用率越高
		2. 同一K，曲线明显：分块和合并内存访问未对齐时，需要拉取的次数增加
