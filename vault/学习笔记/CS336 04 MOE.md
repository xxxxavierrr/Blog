1. 架构：
	1. FFN层拆为很多子FFN，路由到其中一个进行训练推理
	2. 由于子FFN是稀疏模型，能在相同FLOPS的情况下训练得到更多参数
	3. 在相同FLOPS情况下，专家越多，loss越低
	4. 专家占用内存，并行+路由导致的系统复杂性
2. 路由函数
	1. 路由架构
		1. top-k
			1. ![[CS336 04 MOE.png|522]]
			2. token选择top-k专家（最常用）
				1. 最佳实践：k=2
				2. topk消耗额外FLOPS
			3. 专家选择top-k token
				1. 每个专家拿到的token相同
			4. 全局分配（保证均衡）
			5. ![[CS336 04 MOE-1.png|516]]
		2. Hash
			1. 简单哈希仍能带来性能提升
		3. RL
			1. 原因：离散决策
			2. 系统稳定性太差，成本太高
	2. top-k原理
		1. x@w（所有专家）
			1. w为门控w，与FFN w不同
			2. 门控w在本地计算
		2. softmax（计算token和专家亲和力），取概率top-k，其余结果清零
			1. 后改为sigmoid
		3. FFN
		4. 结果加权
	3. 专家
		1. 普通FFN：Dff = 4 * Dmodel
		2. 相同计算量下，更小的Dff -> 更多的专家
		3. 共享专家
	4. 训练
		1. 如何保证启用部分专家的情况下能够训练所有的专家
			1. 强化学习
				1. 过于复杂，性能不好
			2. 随机近似
				1. 门控w里加入噪音w，扰动路由选择
				2. 导致专业化程度低，但更全面
			3. 损失均衡
				1. 解决局部最优（一直路由训练表现最好的专家）
				2. ![[CS336 04 MOE-2.png|524]]
					1. N：专家总数
					2. α：超参数权重系数，用来控制辅助损失对总损失的影响强度
					3. f P点积：所有专家的「实际分配占比 × 平均概率占比」之和，负载越不均匀，这个值就越大
					4. f： batch 里真正被送到专家 i 的 token 数量，除以总 token 数
					5. P：整个 batch中的token 经过 Softmax 后，分配给专家 i 的路由概率值
					6. f不可导，负责权重，P可导
					7. 按batch平衡
				3. ![[CS336 04 MOE-3.png|429]]
					1. 动态偏置
		2. ![[CS336 04 MOE-4.png]]
			1. 批量矩阵乘法->对角矩阵乘法->块稀疏矩阵乘法
			2. 天然支持负载不均衡
		3. top-m device routing
			1. 对每个批次token先选择top-m个设备，减少通信开销
		4. 随机性
			1. 启发式路由
			2. 每个专家有处理上限，多余token直接删
		5. 稳定性
			1. z-loss
			2. 后训练防止过拟合
				1. dense和moe交替，只调dense
				2. 使用巨量数据sft
			3. upcycling：dense copy->moe
