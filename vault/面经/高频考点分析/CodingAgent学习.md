亮点：

1. **智能上下文治理（核心竞争力）**自研 Token 精细化管控体系，支持四级预警、动态压缩、比例缓冲、熔断防死循环，完美解决不同 LLM 窗口不一致、上下文爆炸的行业痛点，兼顾性能与成本。
2. **安全合规的执行范式**独创**计划模式**，强制「先调研 → 写方案 → 再执行」，内置只读沙箱，禁止高危操作，从流程上杜绝 AI 盲目修改、误操作风险，是生产级安全设计。
3. **渐进式任务管理系统**从轻量内存待办到**持久化任务图**，支持任务依赖、稳定 ID、级联清理，既能做简单待办，也能管理复杂多步骤工程任务，适配全场景工作流。
4. **标准化 MCP 生态扩展**原生集成 MCP 客户端，支持 stdio/HTTP/SSE 三种协议，自动拉取、封装外部工具，实现插件化能力扩展，不改动核心代码即可无限扩容 AI 能力。
5. **全链路权限与隔离设计**会话隔离、工具权限校验、计划模式权限锁、MCP 调用隔离，多层防护，无越权操作，安全性远高于普通简易 Agent。
6. **性能与鲁棒性拉满**工具结果截断、Token 分层重试、连接缓存、失效锚点清理，全方位避免 LLM 推理变慢、幻觉、死循环，稳定可用。
7. **模块化架构，易扩展易维护**流程管控、任务系统、工具调用、外部扩展完全解耦，模式可自由切换，二次开发、定制化成本极低。

Q1：整个项目，真正发送请求调用大模型的代码在哪？\
A：只在 Step1client.messages.stream({...})所有后续 Step2~Step11 全部是 封装 / 调度 / 业务逻辑，不直接发 LLM 请求。

Q2：大模型为什么会输出 type: "tool\_use"？是谁规定的？

A：\
代码侧：提前定义好工具描述 + 入参 Schema传给大模型；\
模型侧：训练 + Prompt 约束，约定「需要外部操作就输出固定 tool\_use 结构」；\
本地代码：监听这个字段，识别并执行工具。

Q3：整套代码分层关系是什么？\
A：\
Step1：LLM 底层流式客户端（底座）\
Step2：终端多轮对话 REPL\
Step3：基础工具规范 + 单个 Read 工具\
Step4：Agent 核心循环（思考→调工具→回传结果→循环）\
Step5：全套实用工具（读写 / 编辑 / 检索 / 终端命令）\
Step6：动态拼接系统 Prompt（静态规则 + 运行时环境）\
Step7：工具调用权限控制（放行 / 询问 / 拒绝）\
Step8：QueryEngine 会话总管理器\
Step9：对话历史持久化（JSONL 会话记录）\
Step10：项目长期记忆（Markdown 知识库）\
Step11：上下文压缩（解决上下文超长超限）\
会话 & 上下文（你重点疑惑）

Q4：toolContext 从哪里来？内部怎么流转？\
A：\
外部入口主动传入（顶层代码、REPL）；\
存放：cwd工作目录、中断信号、环境变量；\
层层透传给：Agent 循环 → 工具 call 函数 → 文件 / 命令操作。

Q5：「对话历史」和「项目长期记忆」有什么区别？\
A：\
对话历史（Step9）\
存：每一轮一问一答、工具调用记录\
位置：用户目录隐藏文件夹 .easy-agent\
作用：维持本轮会话上下文\
长期记忆（Step10）\
存：项目规范、技术栈、长期知识点\
格式：单个知识点 = 单个 MD 文件\
作用：跨会话、跨重启永久复用

Q6：长期记忆存在哪里？会存到当前代码仓库里吗？\
A：\
存在电脑用户家目录，和你的项目代码完全隔离；\
按项目目录哈希隔离，不同项目记忆互不干扰；\
不会污染本地代码仓库。

Q7：记忆检索 findRelevantMemories 需要调用大模型吗？\
A：不需要，纯本地运算：\
关键词 includes 字符串匹配打分；\
无向量、无 AI、无正则、低成本。\
工具调用 & Agent 循环（新增关键）

Q8：为什么绝对不能拆分 tool\_use 和 tool\_result？\
A：Anthropic 协议强制要求：\
一次工具调用必须成对出现；\
割裂会直接报错、上下文解析失败；\
所以压缩 / 截断时必须做「安全点位校验」。

Q9：微压缩、全压缩 分别做什么？会删消息吗？\
A：\
微压缩（MicroCompact）\
不删除任何消息、不拆工具配对；\
只清空老旧工具返回的大段文本，替换为占位符；\
零成本、不调大模型。\
全压缩（Full Compaction）\
会截断早期历史消息；\
调用大模型生成全局对话摘要；\
只保留最近几条完整上下文。

Q10：微压缩会不会连着工具调用一起删掉 / 拆开？\
A：不会：\
只修改 tool\_result 内容，tool\_use 完整保留；\
所有消息条数、结构、配对完全不变；\
只有全压缩才会删除旧消息。

Q11：大模型什么时候结束回答、停止 Agent 循环？代码在哪？\
A：\
核心判断代码（Step4）：\
js\
if (result.stopReason !== "tool\_use") {\
return { state, usage: result.usage, reason: "completed" };\
}\
逻辑：\
大模型返回 stopReason: "tool\_use" → 还要调用工具 → 继续循环；\
大模型返回其他 stopReason → 不需要工具、已经想好了最终答案 → 直接 return 终止 Agent 循环；\
额外兜底：超出最大轮次 maxTurns 也会强制停止。\
权限 & Prompt & Token

Q12：工具权限模型（Step7）逻辑是什么？\
A：三级策略：\
只读操作（ls/cat/ 读文件）→ 自动放行；\
写入 / 修改操作（写文件 /npm 安装）→ 询问用户；\
高危操作（rm/sudo/git reset）→ 直接拒绝。

Q13：系统 Prompt 为什么要动态组装（Step6）？\
A：每一轮对话实时刷新：\
固定静态：AI 身份、行为约束；\
动态注入：当前目录、系统版本、Git 状态、项目 AGENT.md 规则。

Q14：Token 估算靠什么？需要调大模型吗？\
A：纯本地字符 heuristic 算法：\
文本：4 字符 ≈ 1token\
JSON / 工具数据：2 字符 ≈ 1token\
结合接口真实用量做校准，零消耗。

Q15：长期项目记忆是谁判断要保存的？是大模型主动说要存吗？本地怎么校验？

A：

1. 不是大模型口头说要保存，也不由模型单独决定；
2. 判断逻辑：模型输出符合固定格式的记忆数据 → 交给本地代码做强制校验 → 校验通过才会保存；
3. 本地校验核心：Step10 的 `shouldStoreAsMemory()` 函数，严格校验 4 个条件：
   * 记忆类型必须是 `user/feedback/project/reference` 其中一种
   * 必须包含 `name/description/body` 三个必填字段
   * 描述文字长度 ≤ 200 字符
   * 数据不能为空
4. 最终规则：本地校验不通过，直接拒绝保存，完全由本地代码把关。

Q16：Step12 四个核心问题 + 对应解法

A：

模型窗口不一致：不同模型上下文上限不同 → 独立配置 + 环境变量覆盖，动态获取真实窗口。

小窗口缓冲区失衡：固定阈值只适配大窗口 → 缓冲区按窗口等比例缩放。

压缩反复失败：自动压缩无限重试 → 增加熔断机制，连续失败 3 次自动禁用。

压缩死循环：压缩操作触发二次压缩 → 压缩来源拦截 + 压缩后失效 Token 锚点。



Q17：Step12 四级 Token 预警状态

A：normal 正常 → warning 空间紧张 → error 自动压缩 → blocking 直接禁止请求，分层防护上下文溢出。

Q18：Step12 工具结果截断、输出 Token 分层的作用

A：超大工具内容提前截断，防止撑爆上下文；输出 Token 分两层：日常 8K 省成本，内容截断时自动升级 64K 重试。



Q19：什么是 UsageAnchor？为什么压缩后必须失效？

A：是 Token 估算的基准锚点，提升计算精度；压缩会重构消息列表，旧锚点数据过期，不清理会导致 Token 误判、触发无限压缩。



Q20：Step13 计划模式核心作用

A：安全管控模式，强制先调研、后规划、再执行；仅允许只读工具 / 只读命令，禁止改文件、高危操作，退出后恢复全部权限。



Q21：Step13 / Step14 / Step15 三者核心关系

A：

Step13：流程权限管控层（全局安全限制）；

Step14：轻量内存待办（临时、无持久化、简单过渡方案）；

Step15：持久化任务系统（14 的升级版，文件落地 + 任务依赖）；

联动：13 管流程，14/15 管具体任务。



Q22：Step15 任务 V2 对比 V1 核心升级

A：文件持久化、重启不丢失；稳定唯一任务 ID、支持任务双向依赖；完整 CRUD，删除任务自动级联清理依赖脏数据。



Q23：Step15 blocks 与 blockedBy 含义

A：blocks：当前任务阻塞下游任务；blockedBy：当前任务被上游任务阻塞；用来自动判断任务是否满足执行条件。



Q24：Step16 MCP 核心作用

A：给 AI Agent 接入外部第三方工具，支持 stdio/HTTP/SSE 三种连接；将外部服务工具统一包装为本地工具格式，实现插件化扩展。



Q25：MCP 工具 和 原生内置工具 关键区别

A：内置工具：代码硬写、用户无法新增、数量可控、性能好；MCP 工具：外部服务动态拉取、用户配置即可添加、扩展性强，但工具过多会拖慢 LLM。



Q26：MCP 工具太多为什么会降低 LLM 性能？

A：所有工具描述、参数 Schema，每轮都会塞进系统提示词；工具越多 Token 越高、上下文被挤占，LLM 决策负担变重，容易乱调用、产生幻觉。



Q27：当前 Step16 教学版中，用户对 MCP 的限制能力

A：用户无法改代码、不能单独筛选单个 MCP 工具；只能自主选择「连接哪些 MCP 服务器」，只能通过少连工具量大的服务，避免 LLM 性能下降。
