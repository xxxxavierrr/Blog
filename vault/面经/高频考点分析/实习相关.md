🔹 一、CRM 客户经营管理模块（1–23）

 

1. 你们 CRM 系统几个人做？做什么？

 

整体后端团队5人左右，我全程负责客户经营模块开发维护。

主要工作：三层主体表设计、多源数据同步、MQ&Binlog 数据接入、分布式并发控制、策略模式重构、大模型 Agent 业务落地、代办推送与日报自动化。

 

2. 客户经营模块面向谁？聚合数据做什么？

 

面向车企/渠道代理商、经销商、一线销售。

核心是聚合合同、资金、发票、风控、历史跟进、主机厂多割裂数据源，统一客户档案，解决数据分散、查询低效、客户经营无抓手的问题，支撑销售跟进、风险识别、绩效统计、代办闭环。

 

3. 三张表怎么设计？各存什么字段？

 

- 公司层表：企业主体ID、公司名称、渠道类型、租户权限、主体基础信息，用于租户隔离、权限边界。

- 聚合层CMID表：公司+品牌+销售联合唯一标识，存储跟进归属、代办绑定、日报统计维度、经营标签，是销售最小业务执行单元。

- 账户层表：代理商真实子账号、合同关联、资金往来、业务原始实体数据，承载底层真实业务流水。

 

4. 为什么分三层？一张表不行吗？好处是什么？

 

不行。

业务是代理商多账号模式，一个公司多品牌、多销售、几十上百个子账户。

单表会导致：销售维度无法拆分、跟进记录混乱、日报/绩效/代办无法按人统计、权限无法隔离。

分层好处：业务解耦、权限隔离、统计维度清晰、归属关系稳定、便于数据聚合与拆分治理。

 

5. 为什么代理商多账户不方便管理？

 

代理商线下子账号数量多、杂乱无统一标准，账号归属模糊；

多销售共用一个企业主体，业务混在一起，无法区分谁跟进、谁负责；

数据混杂，没法单独统计个人工作量、客户跟进情况，经营风险无法精准定位到人。

 

6. 为什么日报要绑定第二层，不绑定第三层账户？

 

销售的工作颗粒度是客户经营维度，不是零散子账号。

第三层是碎片化业务账号，无经营属性；

第二层CMID是销售唯一归属单元，绑定后才能统一做跟进统计、日报汇总、代办考核、工作量核算，业务口径统一。

 

7. 数据同步流程是什么？有哪些数据源？

 

数据源：合同、资金、发票、客户档案、主机厂业务数据、风控数据、数仓T+1离线报表。

流程：

业务库Binlog增量监听 → MQ 异步分发 → 消费端数据清洗规整 → 策略分发落地 → 实时增量更新；

凌晨搭配数仓T+1全量同步按月校对，修正数据偏差，保障数据准确。

 

8. 监听 binlog 是什么？为什么监听？

 

Binlog 是 MySQL 二进制日志，记录表新增、修改、删除所有变更事件。

作用：无侵入、低耦合，不用改业务代码就能捕获数据变动；

用来做下游数据同步、缓存更新、异构系统数据打通，实时联动CRM客户数据。

 

9. 有几个 MQ？分别监听什么？

 

拆分多个独立MQ队列，按业务域拆分解耦：

交易资金队列、合同订单队列、客户档案队列、风控异动队列；

不同数据源流量隔离，削峰防雪崩，互不影响，同步稳定性更高。

 

10. 策略模式+工厂模式怎么用？通用逻辑、不同逻辑是什么？

 

整体同步流程通用逻辑完全统一：

加载主体 → 校验客户关系 → 数据规整 → 落地入库。

差异化逻辑下沉到策略：不同数据源的字段适配、来源解析、校验规则不一样。

工厂模式根据业务类型路由对应策略，新增数据源只新增策略类，不改主流程，符合开闭原则。

 

11. 策略接口有哪些方法？

 

统一抽象三层核心方法：

 

1. loadCustomer 加载源数据

2. checkExist 校验关联关系

3. syncData 落地同步写入

 

12. 为什么用分布式锁？冲突场景是什么？

 

核心问题：查询+插入是非原子操作。

多MQ消费者并发消费，多个线程同时查询CMID不存在，同时发起插入，造成重复数据、脏数据。

引入分布式锁，保证同一业务维度串行执行，防止并发重复创建。

 

13. 锁粒度为什么是公司维度？细粒度为什么会死锁？

 

细粒度是 公司+品牌+销售，维度太多、加锁顺序不统一。

不同业务流程加锁先后顺序不一致，会形成循环等待、资源互相持有，直接触发死锁。

公司维度天然业务隔离，冲突范围小、加锁简单，从源头规避死锁。

 

14. 死锁场景具体是什么？

 

任务A 先锁品牌1、再申请品牌2；

任务B 先锁品牌2、再申请品牌1；

双方各自持有对方需要的锁，互相等待释放，无限阻塞，形成标准死锁。

 

15. Redisson 看门狗原理？过期时间？续期机制？

 

看门狗是Redisson内置后台定时任务，不是手动新开线程。

如果业务不指定锁过期时间，默认30s；

每隔10s自动续期、延长锁有效期，防止业务执行时间过长、锁提前释放，导致并发问题。

业务执行完成主动解锁，避免无效续期。

 

16. Redis String 底层 SDS 结构？

 

SDS 是简单动态字符串，区别于原生C字符串：

 

1. 记录长度，O(1)获取字符串长度

2. 预分配空间，减少频繁内存扩容

3. 二进制安全，兼容特殊字符、业务二进制数据

4. 杜绝缓冲区溢出，更稳定安全

 

17. 大模型在 CRM 里做什么？输入输出？

 

输入：客户档案、合同资金静态数据、历史跟进记录、代办任务、附件RAG知识库。

能力输出：

 

1. 每日客户300字经营总结

2. 销售企微沟通话术

3. 待办完成跟进总结

4. 自动汇总生成销售日报

以Agent+RAG+MCP架构，辅助销售轻量化办公。

 

18. 业务价值是什么？给谁用？解决什么痛点？

 

服务一线销售。

痛点：客户数据分散十几个系统、手动查询耗时、不会做客户经营、跟进无内容、每日统计日报重复机械工作。

价值：一键汇总客户信息、智能生成话术、跟进闭环、自动日报，大幅降低重复工作，提升业务规范性与跟进效率。

 

19. 大模型调用怎么防超时、防打挂？用了什么队列、SSE？

 

整体异步化改造：

引入内部异步任务队列消化大模型耗时任务；

前端/客户端基于SSE流式推送，避免同步阻塞超时；

同时加接口限流、超时熔断、失败降级、异常重试，防止大模型服务被打挂、雪崩。

 

20. 模型为什么选 GPT 不选豆包/Gemini？

 

业务偏向长文本总结、专业业务话术生成、结构化输出。

GPT 长文本理解、业务贴合度、输出稳定性、结构化能力更强；

行业落地案例多，容错率高，适合企业级复杂业务场景。

 

21. 怎么评估模型好坏？有哪些维度？

 

五大维度：

 

1. 业务准确性：客户数据、风险信息无错误

2. 内容贴合度：贴合销售实际经营场景

3. 话术可用性：可直接复制企微发送

4. 响应性能：生成耗时、稳定性

5. 容错鲁棒性：残缺数据下正常降级输出

 

22. 客户档案数据来自哪里？数仓 T+1 按月同步？

 

实时数据：业务库Binlog+MQ增量实时同步；

离线海量档案、历史经营数据：依赖数仓T+1日级、按月全量快照同步，做数据补全与校对。

 

23. 代办任务、提醒、飞书推送怎么做？

 

后台统一代办中心，绑定CMID经营单元；

数据异动、待办到期、客户风险触发事件后，调用内部推送服务；

对接飞书机器人webhook，实现定向消息推送、到期提醒、风险告警，保证跟进闭环。

 

 

 

🔹 二、爱奇艺 DMS / 混沌工程（24–34）

 

24. 老系统问题是什么？

 

老旧DMS权限混乱、无统一管控；

研发私自持有线上数据库明文密码；

无SQL审计、无高危拦截、无访问限制；

存在脱库、慢查询、高危写入、数据泄露重大合规风险。

 

25. 为什么重构？核心目标是什么？

 

核心目标：回收线上数据库密码，落地安全合规。

统一权限管控、操作全程审计、高危SQL拦截、访问最小化、消除数据安全隐患，标准化研发数据库访问流程。

 

26. 无密码连接、权限代理如何实现？

 

中间增加代理服务层，研发不持有任何明文密码；

所有数据库连接请求经过代理转发，统一鉴权、校验、拦截；

代理层托管数据库账号密码，对内透明，对外屏蔽底层库信息。

 

27. RBAC 权限怎么接入？

 

接入公司统一RBAC权限体系，按用户、角色、业务线、环境多维度绑定；

不同研发只能访问自己业务线、对应环境的库表，做到权限隔离、最小授权。

 

28. 只读账号怎么创建？存在哪里？

 

提交权限申请后，系统自动批量创建只读账号；

账号信息统一加密存入CMDB配置中心，集中管理、生命周期可管控，禁止外泄。

 

29. 高危 SQL 拦截规则有哪些？

 

性能类：全表扫描、无索引查询、大limit批量查询、for update行锁、in 集合过大；

安全类：批量删改、跨库查询、批量导出语句、敏感字段查询限制。

 

30. 如何判断脱库行为？SQL指纹、limit 拦截？

 

通过AST语法树解析SQL结构；

结合SQL指纹识别批量遍历库表、无边界分页、全量导出特征；

强制限制返回行数、拦截无limit大范围查询，识别批量拖库风险并告警拦截。

 

31. 策略模式怎么抽象多数据库？

 

抽象统一数据库操作顶层接口；

针对 MySQL、PG 等不同数据库方言，实现独立策略实现类；

上层业务无感知，统一调用，屏蔽多数据库语法、驱动、连接差异。

 

32. 连接池为什么用短连接？长连接风险？

 

线上环境优先短连接。

长连接风险：连接长期不释放、连接泄露、权限过期残留、多人共用连接导致权限串扰、安全不可控。

短连接用完即释放，风险更低、更合规、便于审计回收。

 

33. AI 生成 SQL 流程？Prompt 怎么做？

 

注入当前库表元数据、字段注释、业务上下文；

通过结构化Prompt约束语法规范、查询范围、禁止高危操作；

AI生成后，再经过DMS高危规则二次拦截校验，才可执行。

 

34. 混沌工程平台做什么？注入故障目的？

 

负责主动故障注入：CPU爆满、内存溢出、网络延迟、依赖超时、节点宕机。

目的：提前主动暴露系统降级不足、容错短板、容灾弱点，提前加固，提升整体架构稳定性与故障自愈能力。

 

 

 

🔹 三、信控授信系统（35–40）

 

35. 信控是做什么的？额度怎么来？

 

面向B端经销商的赊销风险评估与动态授信。

整合多源风控数据，通过评分卡+风险策略打分，自动计算企业可赊销额度，控制渠道交易坏账风险。

 

36. 数据来源：天眼查/财报/主机厂怎么接入？

 

三方接口标准化接入：

天眼查工商、司法、失信数据；

企业财报经营指标；

主机厂销量、合作履约历史；

统一清洗入数仓，供策略引擎计算使用。

 

37. 策略模式如何分装风险评估逻辑？

 

抽象统一风险评估流程；

不同企业类型、不同风险等级，差异化规则做成独立策略；

工厂动态路由，新增风控规则无需改动主流程，配置化快速上线。

 

38. 评分卡、Json 配置、expr 表达式？

 

风控规则全部配置化、低代码：

通过JSON配置指标阈值、权重；

expr表达式引擎动态解析规则公式，实时计算得分；

依托评分卡模型，加权汇总得出最终风险分数。

 

39. T+1 全量 & 增量同步？

 

日常经营数据、风险异动走增量实时同步；

企业财报、工商这类低频大变更数据，采用T+1全量刷新；

增量保实时，全量保准确，双向互补。

 

40. 数据如何保证准确？

 

1. 多源数据交叉校验

2. 增量+全量双层校对

3. 异常数据拦截、脏数据清洗

4. 定时数据巡检、偏差告警

 

 

 

🔹 四、通用项目问题（41–46）

 

41. 你最有挑战/成就感的项目是什么？

 

懂车帝CRM客户经营模块。

既要理解复杂代理商多层级业务，又要解决高并发数据同步、分布式锁死锁问题，还要落地大模型AI业务，技术+业务综合挑战最大，落地价值最明显。

 

42. 项目难点是什么？怎么解决？

 

1. 多源数据同步一致性：Binlog+MQ异步解耦+全量定时校对

2. 并发重复写入：合理分布式锁粒度，规避死锁

3. 大模型服务不稳定：异步队列+SSE+熔断限流降级

4. 复杂业务扩展：策略模式解耦差异化逻辑

 

43. 你在项目里具体负责哪些？

 

独立负责模块后端开发、表结构设计、数据同步流程搭建、并发问题治理、设计模式优化改造、大模型Agent对接落地、定时任务与消息推送开发，从需求对接、编码到上线全流程跟进。

 

44. 为什么这么设计表/架构/流程？

 

所有设计都是业务驱动：

分层表为了适配代理商多主体管理；

MQ+Binlog为了低耦合实时同步；

分布式锁为了解决并发原子性问题；

策略模式为了长期迭代扩展、降低维护成本。

 

45. 数据一致性怎么保证？

 

实时：MQ消费幂等、唯一键防重、事务控制；

离线：T+1全量数据校对、跨数据源比对；

最终采用最终一致性方案，满足业务允许短暂延迟的场景。

 

46. 高并发、冲突、重复数据怎么解决？

 

1. 唯一索引、业务联合唯一约束兜底

2. 分布式锁控制并发串行化

3. MQ消费幂等、消息去重

4. 写入前置查重、数据校验
