# Redis

1. Redis MemCache区别
   * 数据类型支持：Memcached仅支持简单的键值对存储，而Redis支持更丰富的数据类型，包括字符串、列表、集合、有序集合、哈希等，为开发者提供了更高的灵活性。
   * 持久化：Redis提供了RDB和AOF两种数据持久化机制，可以将内存中的数据保存到磁盘中，保证数据的安全性。Memcached不支持数据持久化，主要设计为纯内存存储，重启后数据会丢失。
   * 分布式支持：Redis自身支持分布式数据存储方案，如Redis Cluster，提供了数据分片和高可用性。而Memcached虽然可以通过客户端分片来实现分布式存储，但不支持原生的数据分片和高可用性。
   * 发布/订阅系统：Redis提供了发布/订阅消息系统，支持消息的广播，这在构建实时消息系统时非常有用。
2. Redis有什么特性，适用于什么场景

核心特性：

```
- 高性能：基于内存操作，单线程模型避免上下文切换。
- 数据结构丰富：支持多种复杂类型，无需序列化。
- 持久化：支持 RDB 和 AOF，保证数据不丢失。
- 高可用：主从复制、哨兵模式、Cluster 集群。
- 原子操作：所有命令都是原子性的，支持事务。
```

典型场景：

```
- 缓存：减轻数据库压力，存储热点数据。
- 分布式锁：利用 SETNX 和 EXPIRE 实现锁机制。
```

3\. Redis里的key怎么存 哈希表负载很高的情况下Redis怎么处理

Redis使用哈希表存储键值对，value使用redisObject统一封装，每个数据库有一个哈希表，通过哈希算法将键映射到哈希槽。<font style="color:rgba(0, 0, 0, 0.85);">使用 MurmurHash2 计算哈希值，链地址法解决冲突，渐进式 rehash 保证扩容效率。</font>当哈希表负载因子>1/<0.1 时，触发扩容/缩容。

4. Redis扩容怎么实现的 怎么理解两个map查

在哈希表扩容时，触发 渐进式 rehash，Redis 同时维护新旧两个哈希表：

```
- 分配更大的哈希表（通常是原表的 2 倍）。
- 旧表用于读取，新表用于写入。
- 渐进式迁移期间，查询会同时访问两个表，确保数据一致性。
- 优点：避免一次性 rehash 导致的线程阻塞，保证服务可用性。
```

5\. Redis单线程（2） 为什么这么设计 为什么内存里就合适，磁盘里就不合适

设计初衷

```
- 单线程无需锁竞争和上下文切换，无需处理多线程并发问题。
- 内存读写速度极快，单线程即可充分利用 CPU。
- 性能瓶颈在网络IO，不在内存操作上。
```

为何不适用于磁盘

```
- 磁盘 IO 慢：单线程处理磁盘 IO 会导致大量时间浪费在等待上，无法发挥多核优势。
- 阻塞风险：磁盘操作会阻塞线程，导致其他请求无法处理。
```

6\. Redis用多线程会有什么问题吗
\- 锁竞争：多线程操作共享数据需加锁，可能导致性能下降。
\- 复杂度增加：需处理线程安全问题，代码维护难度增大。
\- 单线程优势丧失：Redis 的高性能依赖单线程模型，多线程可能反而降低效率（除非是 IO 密集型操作）。
7\. Redis多线程连接怎么理解 老版本不能使用多线程吗

Redis 6.0 引入 多线程 IO，但核心命令执行仍为单线程：

```
- 接收请求、解析协议、发送响应等 IO 操作由多线程处理。
- 命令执行仍由主线程负责，保证原子性。
```

主从Reactor流程：

```
- MainReactor（主线程）通过监听连接建立事件，由 Acceptor 处理连接请求。
- MainReactor 将新连接分配给 SubReactor（子线程）。
- SubReactor 将连接加入监听队列，并创建 Handler 处理后续事件。
- 当事件发生时，SubReactor 调用对应的 Handler 处理。
```

Redis 6.0 前仅支持单线程，需通过以下方式优化：

```
- 部署多个 Redis 实例利用多核 CPU。
- 使用 Cluster 集群分片数据。
```

8\. Redis怎么处理并发client请求

主线程接收请求：通过 IO 多路复用监听客户端连接，将请求放入队列。

**IO 线程读取数据：多个 IO 线程并行从套接字读取请求数据（解析协议）。**

请求队列：客户端请求进入队列，按顺序被主线程处理。

**<font style="color:rgb(0, 0, 0) !important;">IO 线程回写响应</font>\*\*\*\*<font style="color:rgba(0, 0, 0, 0.85) !important;">：IO 线程并行将命令结果写回客户端。</font>**

9. RDB、AOF优缺点

RDB 的优缺点：

优点：

```
    * 紧凑的二进制文件：RDB 文件是紧凑的二进制文件，适合做冷备和全量复制。
    * 对性能影响小：RDB 使用单独的子进程进行持久化，主进程不会进行任何 IO 操作，保证了 Redis 的高性能。
```

缺点：

```
    * 数据丢失风险：RDB 不能实时保存数据，可能会丢失自上次执行 RDB 备份到当前的内存数据。
    * 生成快照时的性能影响：当数据量非常大时，从父进程 fork 子进程进行保存至 RDB 文件时需要一点时间，可能会导致服务暂停数毫秒或数秒。
```

AOF 的优缺点：

优点：

```
    * 数据安全性高：AOF 可以更好地保护数据不丢失，一般每秒执行一次 fsync 操作，最多丢失 1 秒钟的数据。
    * 写入性能高：AOF 日志文件以 append-only 模式写入，没有磁盘寻址的开销，写入性能高。
    * 适合灾难恢复：AOF 日志文件的命令通过可读可理解的方式进行记录，适合做灾难性的误删除紧急恢复。
```

缺点：

```
    * 文件较大：对于同一份数据来说，AOF 日志文件通常比 RDB 数据快照文件更大，恢复速度慢。
    * 写 QPS 低：AOF 支持的写 QPS 会比 RDB 低，因为 AOF 一般会配置成每秒 fsync 一次日志文件。
```

综合使用 AOF 和 RDB：用 AOF 来保证数据不丢失，作为数据恢复的第一选择；用 RDB 来做不同程度的冷备，在 AOF 文件都丢失或损坏不可用的时候，还可以使用 RDB 来进行快速的数据恢复。

10. Redis淘汰策略

Redis 提供 8 种内存淘汰策略（通过 maxmemory-policy 配置）：

```
- noeviction：默认策略，内存不足时拒绝写操作，读操作仍可执行。
- allkeys-lru：从所有键中淘汰最近最少使用（LRU）的键。
- allkeys-random：从所有键中随机淘汰。
- volatile-lru：从设置了过期时间的键中淘汰 LRU 的键。
- volatile-random：从设置了过期时间的键中随机淘汰。
- volatile-ttl：从设置了过期时间的键中淘汰剩余时间最短的键。
- allkeys-lfu（Redis 4.0+）：从所有键中淘汰最不经常使用（LFU）的键。
- volatile-lfu（Redis 4.0+）：从设置了过期时间的键中淘汰 LFU 的键。
```

11\. Redis过期怎么删除 为什么这么设计（2）

Redis 采用 惰性删除 + 定期删除 结合的策略：

```
- 惰性删除：访问某个键时，检查是否过期，若过期则删除。
- 定期删除：每秒执行 10 次（可配置），随机检查一部分键，删除过期键。
```

设计原因：

```
- 避免全量扫描所有键，减少 CPU 消耗。
- 结合惰性删除，保证过期键最终被删除。
```

12\. 为什么Redis一过期就删不合适
\- 性能损耗：实时监控所有键的过期时间需要额外线程，消耗 CPU 资源。
\- 阻塞主线程：删除大量过期键可能导致主线程阻塞，影响服务响应。
\- 实现复杂：需维护精确的定时器，增加系统复杂度。
13\. 如果要一过期就删怎么设计
\- ExpireKeyMap按过期时间存储在ZSet中，定时轮询/添加堵塞队列，对比当前时间戳与ZSet首位时间戳移除已过期数据。
\- <font style="color:rgba(0, 0, 0, 0.85);">使用时间轮高效管理大量移除已过期数据的定时任务。</font>

15. Stream优缺点

优点：高性能读写、轻量级部署

缺点：

消息可靠性、事务性与顺序性较弱：

```
    * 异步持久化风险：默认配置下（fsync=everysec），若 Redis 崩溃可能丢失 1 秒内的消息。
    * 无事务保证：不支持消息处理的原子提交（如 Kafka 的幂等生产者 + 事务）。
    * 不保证严格顺序：在消费组模式下，消息可能被多个消费者并行处理，破坏顺序性。
```

扩展性有限：

```
    * 单节点瓶颈：Redis 是单线程模型，写入吞吐量受限于单个节点（通常 < 20w QPS）。
    * 分片复杂度高：需客户端或代理层（如 Redis Cluster）实现分片，而 Kafka 天然支持分区扩展。
```

消息堆积处理能力差，所有消息保留在内存中，大量堆积可能导致 OOM

高级特性缺失、社区生态不完善

16. Redis hash底层实现

Redis Hash 底层使用 压缩列表 或 哈希表：

```
- 压缩列表：当哈希元素较少且值较小时使用，节省内存。
- 哈希表：当元素数量超过 hash-max-listpack-entries（默认 512）或单个值超过 hash-max-listpack-value（默认 64 字节）时，转为哈希表。
```

18\. Redis集群模式有哪些 怎么选型

集群模式：

```
- 主从模式：一个主节点多个从节点，主写从读，提高读性能。
- 哨兵模式：在主从复制基础上增加哨兵节点，监控主节点状态，自动故障转移。
- 集群模式：原生支持分片，数据分散在多个节点，自动故障转移。
```

选型建议：

```
- 读多写少：主从复制或哨兵模式。
- 大数据量：Cluster 集群。
- 高可用性：哨兵或 Cluster。
- 简单场景：单节点或主从复制。
```
