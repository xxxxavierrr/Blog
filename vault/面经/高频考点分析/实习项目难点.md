1. DMS
   * 权限管控
   * 多数据库适配设计
     * 基于策略模式封装标准化接口，针对 MySQL、Oracle 等不同数据库类型，开发独立的 SQL 生成策略类；通过接口统一调用，底层自动适配目标库的语法规则。
   * 连接池设计
     * 采用 HikariCP 连接池 + 懒加载机制，仅在数据库有查询请求时创建连接池，闲置超时后自动回收；同时设置全局连接池数量上限与并发控制，超过上限的请求进入等待队列。
   * 高危SQL拦截规则与方法
     * 数据量限制：拦截返回行数过多的查询，避免海量数据拉取导致服务卡顿；禁止循环中批量获取大量数据，防止数据溢出。
     * 全表扫描拦截：严格禁止无 WHERE 条件的查询、无 WHERE 条件的 COUNT 统计，杜绝全表扫描拖垮数据库。
     * 索引失效拦截：禁止全字符串模糊匹配（%xxx%）、不等于（!=/<>）查询、负向条件（NOT IN/NOT LIKE）查询，避免索引失效引发性能问题。
     * 锁风险拦截：禁止使用FOR UPDATE等行锁语法，规避数据库锁表、业务阻塞风险。
     * 语法与参数规范：建议合理使用ORDER BY排序；限制查询入参数量，防止参数过多导致执行计划异常。
   * 智能生成SQL怎么设计Prompt与工作流
     * 构建 “预处理 - 生成 - 校验 - 优化” 闭环流程。预处理阶段清洗自然语言查询、替换 SQL 关键字；生成阶段通过 Prompt Engineering 约束模型输出低性能 SQL；校验阶段调用高危 SQL 拦截接口，对异常 SQL 返回报错信息并触发模型重新生成。
2. 混沌工程
   * 整体架构模式\
     平台采用管理控制台 + 分布式 Worker 探针架构。\
     管理端：负责实验配置、任务审批、定时启停、故障策略编排、统一管控；\
     目标实例端：部署 Chaosblade 探针（Agent），监听专属端口，接收平台下发指令，本地执行故障动作；\
     任务流程：故障实验需先审批，审批通过变为待启动状态，支持手动 / 定时自动执行，保障线上操作合规可控。
   * 三类故障执行方案\
     线上微服务环境下，故障注入分为三种实现方式：\
     数据库层故障\
     通过工作流调度，直接操作数据库实例，实现 MySQL 重启、连接异常等故障模拟。\
     容器 / 应用命令级故障\
     依托 K8s 环境能力，远程下发部署、启停、命令执行等操作，完成应用层故障演练。\
     自定义脚本类故障（核心难点）\
     针对复杂网络协议类故障，需要远程连接目标实例，传输、部署自定义 Shell 脚本，通过脚本执行定制化故障逻辑，DNS 协议篡改就是典型代表。
   * 核心难点：自定义 DNS 故障注入\
     需要打通实例远程访问权限，提前在实例侧放行探针服务与 22 端口；\
     远程上传自定义脚本至目标机器，手动执行脚本命令完成故障启停；\
     脚本底层实现DNS 解析劫持：仅放行指定 IP 正常解析，其余域名全部错误解析，模拟域名解析异常故障；\
     相比于框架自带故障，自定义脚本方案开发量大、权限依赖强、环境适配复杂，是本次开发最大难点。
   * Chaosblade 主流故障底层原理\
     系统资源故障（CPU / 内存 / 磁盘）：通过死循环、持续申请内存、生成垃圾文件等进程级方式实现；\
     网络故障（延迟、丢包、断连）：依赖 Linux 原生 tc、iptables 内核网络能力控制；\
     JVM 应用故障：基于 Java Agent + 字节码增强，无侵入修改方法逻辑，实现延迟、异常、返回值篡改；\
     探针核心作用：部署在每台业务实例，负责接收远端指令、本地执行故障、心跳上报、统一恢复，防止故障扩散与残留。
