1. 如何防止大模型输出错误

系统prompt：

```latex
你只能严格基于【给定参考数据】进行总结、整合、润色。
1. 禁止编造客户信息、产品政策、业务规则、价格、时效等任何事实内容；
2. 参考数据为空的内容，如实写「无」，不要自行补充；
3. 禁止输出多余解释、markdown、客套话、前言后语；
4. 所有内容仅做客观整理，不夸大、不承诺、不主观臆断；
5. 严格遵守后续指定的输出格式与字数要求。
```

子agent巡查：

```latex
你是内容安全&事实巡检员，仅做风险排查，不要修改原文、不要主观打分。
基于【原始参考数据】和【AI生成结果】，只检查以下4项：
1. 是否编造了客户信息、产品政策、价格、业务规则等不存在的内容；
2. 是否出现夸大承诺、违规话术、不合规表述；
3. 是否内容空洞、逻辑矛盾、与原始客户信息严重不符；
4. 是否存在无效废话、偏离业务主题内容。

输出要求：
- 无任何风险：只输出「正常」
- 有风险：列出具体风险点，不做删除、不做否定结论
```

2. 业界 三种标准风险处理逻辑
+ 方案 1：纯约束 + 硬规则拦截
    - 核心手段 前置防幻觉提示 + JSON Schema 强输出 + 代码 / 工作流硬规则校验（字数、非空、JSON 格式、必填项校验） 
    - 处理逻辑 格式 / 字数 / 结构类错误：允许单次自动重生成； 事实类错误：靠数据源锁定 + 前置 Prompt 从源头杜绝； 无任何 AI 巡检、无二次大模型调用。 
    - 适用场景 企业销售、日常办公、CRM 回填、日报总结； 
    - 优点 极简、高并发稳定、低成本、无死循环风险。 
+ 方案 2：轻量巡检 + 风险标记
    - 核心手段 在方案 1 基础上，增加轻量巡查子 Agent后置扫描 
    - 处理逻辑 内容合格：直接放行，写入业务系统； 检测出事实 / 合规风险：只打标签、只预警； 绝不自动驳回、绝不 AI 自行修改、不触发重写； 风险数据单独进入人工复核池兜底。 
    - 适用场景 需要基础合规管控、内容对外输出的企业业务。 
+ 方案 3：AI 自检 + 自动纠错 + 人工终审
    -  核心手段 多智能体联动：巡检模型 + 纠错模型，多层校验 处理逻辑 巡检识别风险 → 将问题点反向传给主模型； 
    - 强制要求原模型自动修改、多轮重试生成； 机器校验通过后，再进入人工最终审核。 
    - 适用场景 金融、医疗、法务、合同、政务等强监管行业；
    - 致命缺点 容易幻觉叠加、无限重试死循环、并发承载力差、成本翻倍。 
+ 最终关键结论
    - 销售 Agent禁止使用方案 3：定时高并发 + CRM 归档数据，不允许 AI 自主篡改业务内容； 
    - 最优落地组合：方案 1 必选 + 方案 2 可选开关，既保证准确性、又保证并发稳定、架构最轻； 
    - 统一铁规：格式问题可自动重跑，事实问题只标记、不 AI 自改。
3. RAG落地方案
+ 本业务仅早间话术生成场景使用 RAG，复盘、日报流程全部禁用 RAG，收缩使用边界、降低出错风险。
+ RAG 只保留两类核心内容：标准化触达话术模板、内部系统名词及线索申报等关键业务规则，精简内容从源头减少大模型幻觉。
+ 知识库由管理员统一维护，业务规则、话术更新直接覆盖旧内容，不保留冲突文档，定期清理过期、冗余资料。
+ 采用轻量化迭代方式，以一线反馈和业务规则变更为依据，只做增量补充与内容替换，更新后小范围测试再全量落地。
+ 格式错乱、内容不规范等问题依靠 JSON Schema 和前置约束 Prompt 解决，非规则类问题不改动 RAG，长期降低维护成本。
